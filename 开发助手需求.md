# 安卓开发助手 (Android Dev Assistant) - 产品需求文档

> 版本: v0.2 | 状态: 已确认 | 日期: 2026-07-09

---

## 1. 产品概述

### 1.1 产品定位
面向安卓开发者的智能辅助工具，通过 Web 界面以文字对话方式交互，帮助开发者快速了解设备与应用状态，并协助排查卡顿、内存泄漏等性能问题。

### 1.2 目标用户
- 主要用户：安卓开发者（本人及团队成员）
- 使用场景：日常开发调试、性能优化、问题排查

### 1.3 核心价值
- **一站式诊断**：无需在多个工具之间切换（AS Profiler / Systrace / logcat / adb shell），一个界面获取关键信息
- **自然语言交互**：用中文描述问题，Agent 自动执行对应的诊断命令并解读结果
- **实时可视化**：右侧面板持续展示设备和目标应用的运行状态

### 2.2 右侧信息面板

#### 2.2.1 设备概览
| 指标 | 数据来源 |
|------|---------|
| 设备型号 / 品牌 | `adb shell getprop ro.product.model` |
| Android 版本 / API Level | `adb shell getprop ro.build.version.release` / `sdk` |
| CPU 架构 / 核心数 | `adb shell getprop ro.product.cpu.abi` / `cat /proc/cpuinfo` |
| 总 RAM / 可用 RAM | `adb shell dumpsys meminfo` / `cat /proc/meminfo` |
| 存储总量 / 可用 | `adb shell df -h /data` |
| 屏幕分辨率 / DPI | `adb shell wm size` / `wm density` |
| 电池电量 / 温度 | `adb shell dumpsys battery` |
| 当前网络类型 | `adb shell dumpsys connectivity` |
| 已连接设备列表 | `adb devices` |

#### 2.2.2 目标应用信息
| 指标 | 数据来源 |
|------|---------|
| 应用包名 | 用户指定或自动检测前台应用 |
| 版本号 / VersionName | `adb shell dumpsys package <pkg>` |
| 目标 SDK / 最低 SDK | 同上 |
| 进程 PID | `adb shell pidof <pkg>` 或 `ps -A` |
| 当前 Activity | `adb shell dumpsys activity activities` |
| APK 路径 / 大小 | `adb shell pm path <pkg>` |
| 权限列表 | `adb shell dumpsys package <pkg>` |
| 签名信息 | `adb shell dumpsys package <pkg>` |

#### 2.2.3 实时性能指标（刷新频率：2秒）

> ⚠️ **注意**：`dumpsys meminfo` 等命令单次执行约 1-2 秒，若上一次请求未返回，后端自动跳过当次轮询，避免命令堆积。

| 指标 | 数据来源 |
|------|---------|
| 内存占用 (PSS/RSS/VSS/USS) | `adb shell dumpsys meminfo <pkg>` |
| Java Heap 使用 / Native Heap | 同上 |
| 当前 FPS / 帧渲染时间 | `adb shell dumpsys gfxinfo <pkg>` |
| Janky frames 比例 | 同上 |
| CPU 占用率 | `adb shell top -n 1 -p <pid>` |
| 当前 Activity 线程数 | `adb shell ps -T -p <pid>` |
| GC 事件统计 | `adb shell dumpsys <pkg> GC` 或 logcat 过滤 |
| ANR 检测 | `adb shell dumpsys activity activities` |

#### 2.2.4 历史趋势图

> **设计理由**：内存泄漏的核心特征是 PSS 随时间持续上涨，卡顿往往是间歇性的。没有趋势图，开发者很难肉眼判断是否存在泄漏或定位卡顿触发点。

| 图表 | 数据来源 | 说明 | 时间窗口 |
|------|---------|------|---------|
| 内存趋势 (PSS) 折线图 | 2.2.3 实时数据缓存 | Y轴 PSS(MB)，X轴时间，异常上涨时高亮警示 | 最近 5 分钟 |
| Java Heap 趋势 | 同上 | Java Heap 分配量变化，关注 GC 后是否回落 | 最近 5 分钟 |
| Native Heap 趋势 | 同上 | Native 内存增长，典型泄漏信号 | 最近 5 分钟 |
| FPS 趋势 | 同上 | 绿色(≥60) / 黄色(30-60) / 红色(<30)，快速定位掉帧区间 | 最近 5 分钟 |
| CPU 趋势 | 同上 | 总使用率 + 应用进程占比 | 最近 5 分钟 |

趋势数据在浏览器内存中维护（环形缓冲区，数据量可控），页面刷新后清空。

### 2.3 左侧对话面板

#### 2.3.1 对话能力
- 自然语言问答，中文交互
- 上下文记忆（多轮对话）
- Markdown 格式回复（表格、代码块、列表等）
- 工具调用过程透明可见（展示 Agent 正在执行的 ADB 命令）

#### 2.3.2 典型对话场景

**场景 1：设备/应用信息查询**
```
用户: "当前连接了什么手机？"
Agent: [执行 adb devices] → 展示设备列表

用户: "这个手机的配置怎么样？"
Agent: [执行多个 getprop + 系统命令] → 结构化展示设备配置

用户: "微信占了多少内存？"
Agent: [执行 dumpsys meminfo com.tencent.mm] → 展示 PSS/RSS/Native/Java Heap
```

**场景 2：卡顿排查**
```
用户: "帮我看看微信最近有没有卡顿"
Agent:
  1. 获取 gfxinfo 帧数据 → 分析 Janky frames
  2. 检查主线程 CPU 占用
  3. 检查 GC 频率是否异常
  4. 检查内存是否接近上限 → 给出综合诊断结论

用户: "帮我抓一个系统 trace 看看"
Agent: [启动 perfetto 或 atrace] → 5秒后停止 → 提示 trace 文件位置
```

**场景 3：内存优化**
```
用户: "分析下目标应用的内存情况"
Agent:
  1. 获取 dumpsys meminfo → 展示 PSS 分解
  2. 检查是否存在内存泄漏特征（如 Activity 数量异常增长）
  3. 分析 Java Heap 各代占用比例
  4. 给出优化建议

用户: "帮我导出 hprof 分析下"
Agent: [执行 am dumpheap → 拉取 hprof → 提供简要分析或提示用 MAT 打开]
```

**场景 4：问题诊断**
```
用户: "应用闪退了，帮我看看什么原因"
Agent:
  1. 抓取最近的 logcat crash 日志
  2. 解析堆栈信息
  3. 定位崩溃原因和代码位置
  4. 给出修复建议
```

### 2.4 高级功能（L3 - 无障碍应用）

#### 2.4.1 自研无障碍 APK 能力
当用户选择安装配套的无障碍应用后，可额外获得：

| 能力 | 说明 |
|------|------|
| 实时 UI 层级获取 | 无需通过 `uiautomator dump`，无障碍服务可直接获取 |
| 窗口切换事件 | 实时感知前台应用和 Activity 变化 |
| 触摸事件追踪 | 记录 Touch 事件坐标、耗时 |
| 系统通知监听 | 获取 Notification 内容 |
| 剪贴板访问 | 读取/写入剪贴板 |
| 应用内 WebView 检测 | 检测 WebView 加载状态 |

#### 2.4.2 通信方案
无障碍 APK 与 Mac Agent 之间的通信通道（二选一或共存）：
- **USB 通道**（推荐）：通过 `adb forward` 端口转发 + Socket 通信，安全稳定
- **WiFi 通道**：通过 HTTP/WebSocket，适合无线调试场景

## 4. 技术方案

### 4.1 技术栈
| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 前端 | Vue 3 (CDN 引入，无需构建工具) | 保持轻量，单页应用 |
| 前端图表 | Chart.js (CDN) | 历史趋势折线图 |
| 后端 | Python FastAPI | 与 LangChain 集成方便 |
| AI 框架 | LangChain + 可切换 LLM | 见 4.1.1 |
| 实时通信 | SSE (Server-Sent Events) | 右侧面板2秒推送，比 WebSocket 更轻量 |
| 手机通信 | ADB + subprocess | 沿用现有方案 |
| 数据解析 | 正则 + 结构化解析器 | 解析 dumpsys 等文本输出 |

#### 4.1.1 LLM 模型切换设计

支持通过配置文件切换模型，一期支持以下模型：

| 模型 | 配置项 | 说明 |
|------|-------|------|
| DeepSeek | `DEEPSEEK_API_KEY` | 现有方案，性价比高 |
| OpenAI 兼容接口 | `OPENAI_API_KEY` + `OPENAI_BASE_URL` | 可接入任何 OpenAI 兼容 API（如本地 ollama、vLLM 等） |

配置文件 `.env` 示例：
```bash
# LLM 提供商: deepseek | openai_compatible
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat

# DeepSeek
DEEPSEEK_API_KEY=sk-xxx

# OpenAI 兼容（LLM_PROVIDER=openai_compatible 时生效）
OPENAI_API_KEY=sk-xxx
OPENAI_BASE_URL=https://api.openai.com/v1
```

切换逻辑在 `agent/llm.py` 中，根据 `LLM_PROVIDER` 初始化对应的 ChatModel。
